# Phase 1 — UI Design Contract

> Visual and interaction contract for Phase 1: Dify Agent API Integration.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Phase classification: BACKEND ONLY**

Phase 1 creates the `/api/brief/chat` API route that proxies messages to Dify's `/chat-messages` endpoint with SSE streaming. There are zero frontend components, pages, or visual changes in this phase. All UI work begins in Phase 2 (Brief Page UI).

This contract documents the existing design system state for downstream consumers and defines the API response contract that Phase 2's UI will consume.

## Copywriting Contract

Phase 1 has no user-facing UI. The following defines the **API response contract** that Phase 2's chat UI will render.

| Element | Copy / Contract |
|---------|-----------------|
| Primary CTA | Not applicable (no UI in this phase) |
| Empty state heading | Not applicable (deferred to Phase 2) |
| Empty state body | Not applicable (deferred to Phase 2) |
| Error: Dify API key missing | "Chat service is not configured. Please contact the administrator." |
| Error: Dify API unreachable | "Unable to reach the chat service. Please try again in a moment." |
| Error: Dify rate limit | "Too many requests. Please wait a moment before sending another message." |
| Error: generic Dify error | "Something went wrong with the chat service. Please try again." |
| Destructive confirmation | Not applicable (no destructive actions in this phase) |

**Error response format** (returned by `/api/brief/chat`):
```json
{
  "error": "Human-readable error message from table above",
  "code": "DIFY_NOT_CONFIGURED | DIFY_UNREACHABLE | DIFY_RATE_LIMIT | DIFY_ERROR",
  "status": 500 | 502 | 429 | 502
}
```

Phase 2 will render these error messages in the chat panel. The copy is defined here so it is consistent across phases.

## API Response Contract (Phase 1 specific)

This section replaces the visual component inventory for this backend-only phase. It defines what the SSE stream returns to the client, which Phase 2's UI must render.

### Success SSE events

**Message chunk** (streamed, multiple per response):
```
data: {"answer": "incremental text chunk", "conversation_id": "uuid", "message_id": "uuid"}
```

**Stream end** (one per response):
```
data: {"event": "done", "conversation_id": "uuid"}
```

**Error during stream:**
```
data: {"event": "error", "message": "Something went wrong with the chat service. Please try again."}
```

### Client request format

```json
{
  "message": "user's message text",
  "conversation_id": "" // empty string for first message, UUID for subsequent
}
```
